Pocketbyte games live or die by how responsive they feel. The SDK's input API gives you frame-accurate button state — whether a button was just pressed this frame, has been held down for several frames, or was just released — so you can write tight, predictable controls for any kind of game. This guide walks you through the full input lifecycle and explains how the swappable module system affects your code.

## Polling input each frame

Before you query any button state, call `pocketbyte_input_update()` once at the very start of your frame. This call snapshots the raw hardware state and computes the pressed/held/released transitions. If you forget it, every query function will return stale data from the previous frame.

```c
void app_main(void)
{
    pocketbyte_init();

    while (1) {
        pocketbyte_input_update(); /* always first */

        /* query buttons here */
    }
}
```

Call `pocketbyte_input_update()` exactly once per frame — not multiple times, and not inside an interrupt handler.

## Available buttons

The SDK exposes eight logical buttons through the `POCKETBYTE_BTN_*` constants. Every input module maps its physical controls to these constants, so your game code never changes when you swap a module.

| Constant | Description |
|---|---|
| `POCKETBYTE_BTN_UP` | D-pad / joystick up |
| `POCKETBYTE_BTN_DOWN` | D-pad / joystick down |
| `POCKETBYTE_BTN_LEFT` | D-pad / joystick left |
| `POCKETBYTE_BTN_RIGHT` | D-pad / joystick right |
| `POCKETBYTE_BTN_A` | Primary action button |
| `POCKETBYTE_BTN_B` | Secondary action / cancel button |
| `POCKETBYTE_BTN_START` | Start / pause |
| `POCKETBYTE_BTN_SELECT` | Select / menu |

## Button state functions

The SDK provides three query functions. Each accepts a `POCKETBYTE_BTN_*` constant and returns a `bool`.

### `pocketbyte_button_pressed(btn)` — first frame of a press

Returns `true` on the single frame the button transitions from not-pressed to pressed. Use this for one-shot actions like jumping, firing, or confirming a menu selection.

```c
if (pocketbyte_button_pressed(POCKETBYTE_BTN_A)) {
    player_jump();          /* fires once per tap, not every frame */
}

if (pocketbyte_button_pressed(POCKETBYTE_BTN_START)) {
    toggle_pause_menu();
}
```

### `pocketbyte_button_held(btn)` — every frame while held

Returns `true` on every frame the button is held down, including the first frame. Use this for continuous actions like movement, scrolling, or charging an attack.

```c
if (pocketbyte_button_held(POCKETBYTE_BTN_LEFT))  player.x -= PLAYER_SPEED;
if (pocketbyte_button_held(POCKETBYTE_BTN_RIGHT)) player.x += PLAYER_SPEED;
if (pocketbyte_button_held(POCKETBYTE_BTN_UP))    player.y -= PLAYER_SPEED;
if (pocketbyte_button_held(POCKETBYTE_BTN_DOWN))  player.y += PLAYER_SPEED;
```

### `pocketbyte_button_released(btn)` — first frame after release

Returns `true` on the single frame the button transitions from pressed to not-pressed. Use this for actions that trigger when the player lets go — releasing a charged shot, ending a drag, or confirming a held action.

```c
if (pocketbyte_button_released(POCKETBYTE_BTN_B)) {
    release_charged_shot(&player);
}
```

## Combining states for nuanced controls

You can combine all three functions in the same frame. A common pattern for a platformer is to read `pressed` for the jump trigger and `held` to extend jump height while the button stays down:

```c
static void handle_player_input(player_t *p)
{
    pocketbyte_input_update();

    /* horizontal movement — continuous */
    if (pocketbyte_button_held(POCKETBYTE_BTN_LEFT))  p->vx = -WALK_SPEED;
    else if (pocketbyte_button_held(POCKETBYTE_BTN_RIGHT)) p->vx =  WALK_SPEED;
    else p->vx = 0;

    /* jump — trigger once */
    if (pocketbyte_button_pressed(POCKETBYTE_BTN_A) && p->on_ground) {
        p->vy = -JUMP_FORCE;
        p->on_ground = false;
    }

    /* variable jump height — hold A to float higher */
    if (pocketbyte_button_held(POCKETBYTE_BTN_A) && p->vy < 0) {
        p->vy -= JUMP_HOLD_GRAVITY;
    }

    /* pause */
    if (pocketbyte_button_pressed(POCKETBYTE_BTN_START)) {
        game_toggle_pause();
    }
}
```

## Swappable input modules

Pocketbyte's input modules are physically interchangeable. A game controller module, a keypad module, and a joystick module all expose the same eight `POCKETBYTE_BTN_*` constants to your code. The SDK detects which module is attached at boot and configures the input subsystem automatically — you don't need to write any detection logic yourself.

<Note>
Do not make assumptions about the physical layout or the number of buttons present on the currently attached module. Not every module includes every button. If a physical button is absent on the attached module, its corresponding `POCKETBYTE_BTN_*` constant will always return `false`. Design your core game loop around the buttons you know will be available, and treat anything else as optional.
</Note>

## A complete input example

```c
#include "pocketbyte/pocketbyte.h"
#include "freertos/FreeRTOS.h"
#include "freertos/task.h"

typedef struct {
    int x, y;
    bool is_jumping;
} player_t;

static player_t player = { .x = 64, .y = 32, .is_jumping = false };

static void process_input(void)
{
    pocketbyte_input_update();

    /* movement */
    if (pocketbyte_button_held(POCKETBYTE_BTN_LEFT))  player.x -= 2;
    if (pocketbyte_button_held(POCKETBYTE_BTN_RIGHT)) player.x += 2;

    /* one-shot jump */
    if (pocketbyte_button_pressed(POCKETBYTE_BTN_A) && !player.is_jumping) {
        player.is_jumping = true;
    }

    /* cancel / go back on B released */
    if (pocketbyte_button_released(POCKETBYTE_BTN_B)) {
        /* return to title screen, etc. */
    }
}

void app_main(void)
{
    pocketbyte_init();

    while (1) {
        process_input();
        /* update and draw ... */
        vTaskDelay(pdMS_TO_TICKS(16));
    }
}
```

<Tip>
As a rule of thumb: use `pocketbyte_button_pressed()` for anything that should happen once per tap (jumps, menu confirms, shooting), and use `pocketbyte_button_held()` for anything that should repeat while the button stays down (walking, scrolling, charging). Mixing them up is the most common source of sluggish or twitchy controls.
</Tip>
